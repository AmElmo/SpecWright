# Core Concepts

## Specification-Driven Development (SDD)

SDD is a development methodology where every feature starts with a structured specification before any code is written. Instead of jumping into implementation from a vague idea, SDD ensures clarity at every level.

## The Four Phases

SpecWright implements SDD through four sequential phases. Each phase builds on the previous one and has validation prerequisites.

### 1. PM Phase

The product manager phase defines **what** to build:

- Problem statement and goals
- User stories and personas
- Acceptance criteria
- Scope boundaries

### 2. Designer Phase

The designer phase defines **how it looks and feels**:

- UI/UX specifications
- Component hierarchy
- User flows and interactions
- Design constraints

### 3. Engineer Phase

The engineer phase defines **how to build it**:

- Technical architecture
- API contracts and data models
- Implementation approach
- Dependencies and risks

### 4. Breakdown Phase

The breakdown phase creates **actionable work items**:

- Task decomposition
- Dependency ordering
- Scope-appropriate chunks
- Ready for AI coding tools or human developers

## File-Based State

SpecWright uses a file-based approach — all project state lives in the `outputs/` directory as JSON and Markdown files. No database required. This makes specs portable, version-controllable, and easy to inspect.

## AI Integration

The structured output from SpecWright is designed to be fed into AI coding assistants like Cursor, GitHub Copilot, or any LLM-powered tool. Well-structured specs produce dramatically better AI-generated code.
